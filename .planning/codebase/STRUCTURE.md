
## Directory Purposes

**`.codex/`:**
- Purpose: Codex-native distribution of GSD assets
- Contains: `skills/gsd-*`, `agents/`, `get-shit-done/`, `config.toml`, managed file manifest
- Key files: `.codex/config.toml`, `.codex/gsd-file-manifest.json`
- Subdirectories: `skills/`, `agents/`, `get-shit-done/{bin,commands,references,templates,workflows}`

**`.claude/`:**
- Purpose: Claude Code distribution of the same GSD toolkit
- Contains: slash commands, hooks, agents, mirrored toolkit resources
- Key files: `.claude/commands/gsd/*.md`, `.claude/gsd-file-manifest.json`, `.claude/package.json`
- Subdirectories: `commands/gsd/`, `agents/`, `hooks/`, `get-shit-done/`

**`.opencode/`:**
- Purpose: Opencode distribution of the toolkit
- Contains: command definitions, hooks, agents, mirrored toolkit assets
- Key files: `.opencode/package.json`
- Subdirectories: `command/`, `agents/`, `hooks/`, `get-shit-done/`

**`.planning/`:**
- Purpose: Mutable project planning state
- Contains: Generated and hand-authored planning docs
- Key files: `.planning/codebase/*.md`
- Subdirectories: `codebase/` currently exists; more milestone/phase files are expected later

**`CodexBar/`:**
- Purpose: Main product repo analyzed in this workspace
- Contains: Swift package, macOS app, CLI, provider engine, scripts, docs, tests, CI config
- Key files: `Package.swift`, `package.json`, `README.md`, `.swiftformat`, `.swiftlint.yml`
- Subdirectories: `Sources/`, `Scripts/`, `Tests/`, `TestsLinux/`, `.github/`, `docs/`

**`ubuntu-extension-analysis/`:**
- Purpose: Standalone design/reference docs for the future Ubuntu extension effort
- Contains: Analysis of `CodexBar`, providers, and Ubuntu direction
- Key files: `README.md`, `codexbar-project-analysis.md`, `provider-analysis.md`, `ubuntu-extension-direction.md`
- Subdirectories: None currently

## Key File Locations

**Entry Points:**
- `.codex/config.toml` - Root agent configuration and hook bootstrap for Codex
- `.claude/commands/gsd/*.md` - Command entry surface for Claude Code
- `CodexBar/Sources/CodexBar/CodexbarApp.swift` - macOS app entry point

- `.github/copilot-instructions.md` - Copilot-specific behavior guidance
- `.codex/gsd-file-manifest.json` - Managed file inventory for Codex distribution

**Core Logic:**
- `CodexBar/Sources/CodexBarCore/` - Shared provider engine and domain logic
- `CodexBar/Sources/CodexBar/` - macOS shell and app-facing state/UI
- `CodexBar/Sources/CodexBarCLI/` - CLI commands and output flow
- `.codex/get-shit-done/workflows/` - Root workflow definitions used by GSD commands

**Testing:**
- `CodexBar/Tests/CodexBarTests/` - Main Swift test suite
- `CodexBar/TestsLinux/` - Linux-only platform gating tests

**Documentation:**
- `ubuntu-extension-analysis/` - Project-specific research docs at the root
- `CodexBar/docs/` - Product-level documentation inside the nested repo
- `.codex/get-shit-done/templates/` - Planning and workflow templates

## Naming Conventions

**Files:**
- `kebab-case.md` for commands, workflows, skills, and many documentation assets
- `PascalCase.swift` for Swift types and test files in `CodexBar/`
- `UPPERCASE.md` for prominent project docs such as `README.md`, `CHANGELOG.md`, `LICENSE`

**Directories:**
- Hidden dot-directories for tool ecosystems at the workspace root
- `PascalCase` target directories inside `CodexBar/Sources/`
- Plural collection directories such as `Sources/`, `Tests/`, `Scripts/`, `templates/`, `workflows/`

**Special Patterns:**
- `gsd-*` naming for commands, skills, and agent-related assets
- `*Tests.swift` for Swift test files
- `Tests/CodexBarTests/` for the main suite and `TestsLinux/` for Linux gating coverage

## Where to Add New Code

**New Planning / Project Definition:**
- Primary docs: `.planning/`
- Reference research: `ubuntu-extension-analysis/`
- Workflow templates: `.codex/get-shit-done/templates/`

**New GSD Capability:**
- Workflow definition: `.codex/get-shit-done/workflows/`
- Codex skill wrapper: `.codex/skills/gsd-*/`
- Claude command surface: `.claude/commands/gsd/`
- Important: treat these directories as managed mirrors; changes should be deliberate and kept consistent across ecosystems

**New `CodexBar` Product Feature:**
- Shared backend logic: `CodexBar/Sources/CodexBarCore/`
- macOS shell/UI logic: `CodexBar/Sources/CodexBar/`
- CLI behavior: `CodexBar/Sources/CodexBarCLI/`
- Tests: `CodexBar/Tests/CodexBarTests/` and `CodexBar/TestsLinux/` when platform gating matters

**Ubuntu Extension Work:**
- Root planning/docs first: `.planning/` and `ubuntu-extension-analysis/`
- Reusable backend target: `CodexBar/Sources/CodexBarCore/` and `CodexBarCLI`
- Do not start by editing the macOS-only AppKit shell unless the task is explicitly about the existing Mac app

## Special Directories

**`.planning/`:**
- Purpose: Mutable workflow state
- Source: Generated and updated by GSD commands
- Committed: Yes, if the team wants planning artifacts versioned

**Hidden tool mirrors (`.codex/`, `.claude/`, `.opencode/`):**
- Purpose: Same toolkit distributed to different agent runtimes
- Source: Installer-managed/mirrored assets, evidenced by file manifests
- Committed: Yes, but edits here require care to avoid drift across mirrors

**`CodexBar/`:**
- Purpose: Nested git repository
- Source: Separate product history and lifecycle
- Committed: Yes, but it is logically distinct from the root meta-workspace
