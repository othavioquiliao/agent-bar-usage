# Milestone v1.1: Provider Reliability

**Status:** Archived 2026-03-26
**Phases:** 6-7
**Total Plans:** 5

## Overview

`v1.1` closed the most urgent post-`v1.0` production blockers by making the backend runtime more reliable under the installed systemd service path and redesigning the GNOME extension for faster scanability. The milestone improved PTY execution for CLI-backed providers, added a first-class Copilot auth flow for Linux installs, captured more user environment during install, and replaced the verbose GNOME popup with a more compact, icon-led surface.

The milestone was archived without a formal milestone audit and without a fresh milestone-scoped `REQUIREMENTS.md` file. Human Ubuntu-host verification was also explicitly waived by user approval from a non-Ubuntu machine, so those verification artifacts remain as follow-up evidence rather than being represented as fully observed production validation.

## Phases

### Phase 6: Provider Reliability

**Goal**: Make all three providers work reliably for any user with minimal friction, both from the CLI and from the systemd background service
**Depends on**: v1.0 archived baseline
**Plans**: 2 plans

Plans:

- [x] 06-01: Replace the `script` PTY wrapper with `node-pty` for Codex and Claude service-mode usage
- [x] 06-02: Add Copilot auth flow, service environment capture, and actionable doctor guidance

**Details:**
Replaced the failing `script`-based PTY path with `node-pty`, added a GitHub Device Flow auth command for Copilot, captured PATH/token/DBus environment into a systemd drop-in at install time, and upgraded doctor output so failed checks point at concrete fix commands.

### Phase 7: GNOME Extension UI Redesign

**Goal**: Redesign the GNOME extension indicator and menu so provider usage is glanceable, icon-led, and progress-bar driven without adding text-heavy diagnostics to the default scan path
**Depends on**: Phase 6
**Plans**: 3 plans

Plans:

- [x] 07-01: Package stylesheet, provider icons, and extension install/runtime asset wiring
- [x] 07-02: Redefine and test the compact indicator/provider-row view-model contract
- [x] 07-03: Render the redesigned indicator/menu UI and pause for GNOME Shell verification

**Details:**
Bundled provider icons and stylesheet assets inside the extension, moved the indicator/menu to a compact aggregate view-model, and delivered a calmer top-bar plus progress-driven provider rows designed for glance-first Ubuntu use.

## Milestone Summary

**Key Decisions:**

- Use `node-pty` as the single PTY execution path for service-mode Codex and Claude fetchers.
- Add a GitHub Device Flow auth command instead of relying on env vars or external tooling for Copilot.
- Treat install-time environment capture and actionable diagnostics as first-class parts of the Ubuntu runtime.
- Package GNOME UI assets inside the extension and keep the top bar aggregate-only while moving detailed diagnostics out of the default scan path.

**Issues Resolved:**

- Removed the most serious systemd/PTTY incompatibility for Codex and Claude provider fetches.
- Added a lower-friction Linux path for Copilot authentication and secret persistence.
- Reduced GNOME menu noise by replacing prose-heavy rows with compact progress-oriented provider summaries.

**Issues Deferred:**

- No formal `v1.1` milestone audit was run before archival.
- No milestone-scoped `REQUIREMENTS.md` existed at archive time; the milestone continued from `PROJECT.md` and roadmap context.
- Real Ubuntu-host verification was waived by explicit user approval from an Arch machine, so production validation evidence remains partial.

**Technical Debt Incurred:**

- Live service verification still depends on local provider configuration, tokens, and a GitHub OAuth client id.
- The Ubuntu host checklist for `v1.1` remains open in `06-HUMAN-UAT.md` as follow-up evidence.
- The next milestone has not been defined yet, so roadmap scope after `v1.1` remains intentionally open.
