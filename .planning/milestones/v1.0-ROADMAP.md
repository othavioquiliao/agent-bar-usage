# Milestone v1.0: Agent Bar Ubuntu

**Status:** Archived 2026-03-26 with accepted gaps
**Phases:** 1-5
**Total Plans:** 15

## Overview

`v1.0` established the Ubuntu product baseline: a Node.js/TypeScript backend, Linux-native configuration and secrets, first-wave provider support, and a GNOME Shell surface. The milestone was archived at user request before the Phase 5 execution artifacts were closed, so delivery-hardening work is explicitly carried into the next milestone as accepted debt rather than being represented as complete.

## Phases

### Phase 1: Backend Contract

**Goal**: Deliver a Linux backend contract that produces stable provider snapshots and refresh metadata independent of any desktop shell
**Depends on**: Nothing (first phase)
**Plans**: 3 plans

Plans:

- [x] 01-01: Scaffold the Node/TypeScript backend workspace and shared snapshot contract
- [x] 01-02: Implement the provider adapter runtime, refresh coordinator, and cache layer
- [x] 01-03: Implement CLI output modes, diagnostics, and backend contract tests

**Details:**
Defined the Linux-facing snapshot envelope, refresh flow, and CLI contract so backend functionality remained independent from GNOME-specific integration points.

### Phase 2: Linux Config & Secrets

**Goal**: Introduce Ubuntu-native configuration and secret management for the Node backend without inheriting Apple-specific assumptions
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:

- [x] 02-01: Design the backend config model and persistence format
- [x] 02-02: Integrate libsecret or equivalent secret-store handling
- [x] 02-03: Wire config and secret resolution into backend provider contexts

**Details:**
Added XDG-backed persistence, provider ordering and enablement controls, and a Linux secret boundary centered on `secret-tool` with explicit fallback behavior.

### Phase 3: First-Wave Providers

**Goal**: Ship the first Ubuntu-viable provider set in the Node backend with reliable fetch paths and clear failure behavior
**Depends on**: Phase 2
**Plans**: 3 plans

Plans:

- [x] 03-01: Implement Copilot Linux flow and snapshot mapping
- [x] 03-02: Implement Codex CLI Linux flow and snapshot mapping
- [x] 03-03: Implement Claude CLI Linux flow plus provider-level failure isolation

**Details:**
Delivered Copilot, Codex CLI, and Claude CLI adapters with deterministic parsing and provider-level isolation so one failing provider would not collapse the overall envelope.

### Phase 4: Ubuntu Desktop Surface

**Goal**: Deliver the first Linux-native desktop surface as a GNOME Shell extension that makes the backend useful in daily Ubuntu workflows
**Depends on**: Phase 3
**Plans**: 3 plans

Plans:

- [x] 04-01: Scaffold the GNOME Shell extension for Ubuntu 24.04.4 LTS
- [x] 04-02: Implement the backend-to-extension bridge and refresh polling model
- [x] 04-03: Build the initial top-bar menu, detail view, and refresh interactions

**Details:**
Built the top-bar surface, detail rows, and refresh interactions on top of normalized backend view models instead of duplicating backend concerns in GJS.

### Phase 5: Delivery & Hardening

**Goal**: Make the Ubuntu v1 installable, diagnosable, and maintainable beyond a local prototype
**Depends on**: Phase 4
**Plans**: 3 plans

Plans:

- [ ] 05-01: Add operational diagnostics and failure-reporting surfaces
- [ ] 05-02: Write Ubuntu installation, setup, and troubleshooting docs for the Node backend and GNOME extension
- [ ] 05-03: Harden packaging, startup behavior, and end-to-end verification

**Details:**
Phase 5 planning, research, and follow-up fixes existed, but its plan summaries were not closed before archival. The remaining delivery-hardening scope was accepted as debt and carried into the next milestone.

## Milestone Summary

**Key Decisions:**

- Use a Node.js/TypeScript backend plus a GNOME Shell extension in GJS for the Ubuntu product.
- Prioritize Copilot, Codex CLI, and Claude CLI as the first Linux provider set.
- Keep provider orchestration and snapshot contracts backend-owned instead of GNOME-specific.
- Defer Cursor and browser-cookie parity work until the Linux contract is stable.

**Issues Resolved:**

- Reframed the product as a Linux-native implementation instead of a Swift/macOS port.
- Established persistent config and Linux secret-storage boundaries for provider execution.
- Delivered a first daily-use GNOME surface for provider visibility and manual refresh.

**Issues Deferred:**

- `OPS-01`, `OPS-02`, and `OPS-03` remained open when the milestone was archived.
- No formal `v1.0` milestone audit was completed before archival.
- Provider reliability from the systemd service path moved into Phase 6 planning.

**Technical Debt Incurred:**

- Phase 5 execution artifacts and summaries remain incomplete for the archived milestone.
- Next-milestone reliability planning started before fresh milestone requirements were redefined.
