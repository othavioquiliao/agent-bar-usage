# Etapa 2: Auth command + install script env capture + doctor improvements

## Goal

Provide zero-friction Copilot authentication, ensure the systemd service inherits the user's environment, and make the doctor command tell users exactly what to do when something is wrong.

## Why this is Etapa 2

Etapa 1 fixes the PTY infrastructure so Codex and Claude can work. This etapa fixes the remaining friction points:
- Copilot needs a token but there's no easy way to provide one
- The systemd service can't see the user's PATH or tokens
- Doctor tells users something is broken but not how to fix it

## Problem Recap

### Copilot: no token, no auto-discovery

The Copilot provider already has a working API fetcher (`copilot-usage-fetcher.ts`). It calls `https://api.github.com/copilot_internal/user` with a GitHub token. The problem is getting that token.

**Current token resolution** (`copilot-token-resolver.ts`):
1. Check env vars: `COPILOT_API_TOKEN`, `GITHUB_TOKEN`, `GH_TOKEN`, `COPILOT_TOKEN`
2. Check secret store: `context.secrets.primary` (from config.json secretRef → secret-tool)
3. If neither found → `copilot_token_missing` error

**What the user decided**: Create `agent-bar auth copilot` using GitHub Device Flow OAuth. Store the token in GNOME Keyring via `secret-tool`. Auto-generate the config file. This is the same flow that `gh auth login` uses — user gets a code, opens browser, authorizes, done.

### Systemd service: minimal environment

The service unit sets only `Environment=NODE_ENV=production`. The user's shell has PATH with `~/.nvm/...`, `~/.local/bin`, tokens, and `DBUS_SESSION_BUS_ADDRESS` (needed for secret-tool to reach GNOME Keyring). None of this reaches the service.

**What the user decided**: The install script captures relevant env vars at install time and writes a systemd drop-in override at `~/.config/systemd/user/agent-bar.service.d/env.conf`.

### Doctor: diagnostic but not actionable

Current `suggested_command` values are verification commands (`which secret-tool`, `codex --version`), not fix commands. Users run doctor, see the problem, but don't know how to fix it.

**What the user decided**: Every check should suggest the actual fix command.

## Implementation Details

### Step 2.1: GitHub Device Flow module

**New file**: `apps/backend/src/auth/github-device-flow.ts`

Implements RFC 8628 (OAuth 2.0 Device Authorization Grant) against GitHub's endpoints.

```ts
export interface DeviceCodeResponse {
  device_code: string;
  user_code: string;
  verification_uri: string;
  expires_in: number;
  interval: number;
}

export interface DeviceFlowResult {
  access_token: string;
  token_type: string;
  scope: string;
}

/**
 * Step 1: Request a device code from GitHub.
 * POST https://github.com/login/device/code
 */
export async function requestDeviceCode(
  clientId: string,
  scope: string,
  fetchFn?: typeof fetch,
): Promise<DeviceCodeResponse>

/**
 * Step 2: Poll for the access token until the user authorizes.
 * POST https://github.com/login/oauth/access_token
 * Handles: authorization_pending, slow_down, expired_token, access_denied.
 */
export async function pollForAccessToken(
  clientId: string,
  deviceCode: string,
  interval: number,
  expiresIn: number,
  fetchFn?: typeof fetch,
): Promise<DeviceFlowResult>
```

**Key details**:
- Both functions accept optional `fetchFn` for unit testing without network
- `pollForAccessToken` handles GitHub-specific error codes:
  - `authorization_pending` → wait `interval` seconds, retry
  - `slow_down` → increase interval by 5 seconds, retry
  - `expired_token` → throw error (code expired, user must restart)
  - `access_denied` → throw error (user denied the request)
- Uses `Accept: application/json` header (GitHub returns JSON when asked)
- `client_id`: needs an OAuth App registered on GitHub for Agent Bar. The command accepts `--client-id` flag for testing/override.

/**

Uses `runSubprocess` from `utils/subprocess.ts` with `input: value` to pipe the secret via stdin. Separate from the existing `SecretToolStore` (which is read-only).

### Step 2.3: Config writer

**New file**: `apps/backend/src/auth/config-writer.ts`

```ts
export interface SecretReference {
  store: "secret-tool";
  service: string;
  account: string;
}

/**
 * Ensure ~/.config/agent-bar/config.json exists and the copilot provider
 * has the given secretRef. Creates the file if it doesn't exist.
 */
export async function ensureCopilotSecretRef(
  configPath: string,
  secretRef: SecretReference,
): Promise<void>
```

**Behavior**:
- If config file doesn't exist → create default config with copilot secretRef
- If config exists but no copilot provider → add copilot provider entry
- If copilot provider exists but no secretRef → add secretRef
- If copilot provider already has correct secretRef → no-op
- Uses the existing config schema from `config-loader.ts`

### Step 2.4: Auth command

**New file**: `apps/backend/src/commands/auth-command.ts`

```ts
export function registerAuthCommand(program: Command): void
```

Registers `agent-bar auth copilot` subcommand. The flow:

```
$ agent-bar auth copilot

  Requesting device code from GitHub...

! Copy this code: AB12-CD34
  Then open: https://github.com/login/device
  Press Enter to open your browser...

  [opens xdg-open https://github.com/login/device]

  Waiting for authorization... (expires in 15 minutes)
  ....

✓ Authenticated!
  Token stored in GNOME Keyring (service=agent-bar, account=copilot).
  Config updated at ~/.config/agent-bar/config.json.
  Service restarted.

  Run "agent-bar doctor --json" to verify.
```

**Internal flow**:
1. Call `requestDeviceCode(clientId, "copilot")`
2. Print `user_code` and `verification_uri`
3. Try `xdg-open <verification_uri>` via `child_process.exec` (silent fail)
4. Wait for Enter (readline)
5. Call `pollForAccessToken(clientId, deviceCode, interval, expiresIn)`
6. Call `storeSecretViaSecretTool("agent-bar", "copilot", access_token, "Agent Bar Copilot")`
7. Call `ensureCopilotSecretRef(configPath, { store: "secret-tool", service: "agent-bar", account: "copilot" })`
8. Run `systemctl --user restart agent-bar.service` (silent fail if not installed)
9. Print success message

**Modified file**: `apps/backend/src/cli.ts`
- Add `import { registerAuthCommand } from "./commands/auth-command.js"`
- Add `registerAuthCommand(program)` after existing command registrations

### Step 2.5: Install script env capture

**File**: `scripts/install-ubuntu.sh`

**Insert BEFORE** the `systemctl --user daemon-reload` line:

```bash
# --- Capture user environment for the systemd service ---
override_dir="${systemd_dir}/agent-bar.service.d"
override_path="${override_dir}/env.conf"
mkdir -p "$override_dir"

env_vars_to_capture=(
  PATH
  GITHUB_TOKEN
  GH_TOKEN
  COPILOT_TOKEN
  COPILOT_API_TOKEN
  ANTHROPIC_API_KEY
  DBUS_SESSION_BUS_ADDRESS
)

{
  echo "[Service]"
  for var_name in "${env_vars_to_capture[@]}"; do
    var_value="${!var_name:-}"
    if [[ -n "$var_value" ]]; then
      echo "Environment=${var_name}=${var_value}"
    fi
  done
} > "$override_path"

echo "Wrote systemd environment override to $override_path"
```

**Why each variable**:
| Variable | Purpose |
|---|---|
| `PATH` | Service needs to find `codex`, `claude`, `node`, `secret-tool` |
| `GITHUB_TOKEN` / `GH_TOKEN` / `COPILOT_TOKEN` / `COPILOT_API_TOKEN` | Copilot API authentication |
| `ANTHROPIC_API_KEY` | Future: Claude API-based fetcher |
| `DBUS_SESSION_BUS_ADDRESS` | Required for `secret-tool` to reach GNOME Keyring daemon |

```ts
{
  id: "systemd-env",
  label: "Systemd env",
  check: async () => {
    const overridePath = path.join(
      process.env.HOME ?? "",
      ".config/systemd/user/agent-bar.service.d/env.conf"
    );
    const exists = await fileExists(overridePath);
    if (!exists) {
      return warn(
        "Systemd environment override is missing. The service may not find CLI tools or tokens.",
        "pnpm install:ubuntu"
      );
    }
    return ok("Systemd environment override is configured.");
  }
}
```

## Risks

| Risk | Mitigation |
|---|---|
| GitHub OAuth App client_id not registered | Use `--client-id` flag for testing; register production app before release |
| secret-tool not installed | Auth command checks for it first and shows `sudo apt install libsecret-tools` |
| DBUS_SESSION_BUS_ADDRESS not set at install time (SSH session) | Doctor `systemd-env` check warns about it; user can re-run `pnpm install:ubuntu` from a GUI session |
| Env override gets stale (user changes PATH later) | Document that `pnpm install:ubuntu` should be re-run after env changes |
| Device Flow code expires (15 min timeout) | Clear error message telling user to retry |
