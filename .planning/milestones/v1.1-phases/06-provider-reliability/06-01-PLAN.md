# Etapa 1: node-pty PTY infrastructure + Codex/Claude fetcher refactor

## Goal

Replace the broken `script -qec` wrapper with `node-pty` so that Codex and Claude providers can fetch usage data from a systemd service (no TTY).

## Why this is Etapa 1

This is the foundation. The `interactive-command.ts` module is shared by both Codex and Claude fetchers. Fixing it unblocks both providers at once. No other work depends on this being done first, but the auth command (Etapa 2) is pointless if the core PTY issue isn't resolved — the extension would still show errors for 2 of 3 providers.

## Problem Recap

### Current flow (broken)

```
codex-cli-fetcher.ts
  └─> runCodexInteractive()
        └─> spawn("script", ["-qec", "codex -s read-only -a untrusted", "/dev/null"])
              └─> FAILS: no controlling terminal in systemd

claude-cli-fetcher.ts
  └─> runInteractiveCommand("claude", [], { input: "y\r\r/usage\r" })
        └─> interactive-command.ts
              └─> spawn("script", ["-qec", "claude", "/dev/null"])
                    └─> FAILS: same reason
```

### Target flow (fixed)

```
codex-cli-fetcher.ts
  └─> runInteractiveCommand(binary, args, { input: "/status\n" })
        └─> interactive-command.ts
              └─> pty.spawn(binary, args, { cols: 120, rows: 30 })
                    └─> WORKS: node-pty creates a real kernel PTY (/dev/pts/N)

claude-cli-fetcher.ts
  └─> runInteractiveCommand("claude", [], { input: "y\r\r/usage\r" })
        └─> interactive-command.ts
              └─> pty.spawn("claude", [], { cols: 120, rows: 30 })
                    └─> WORKS: same mechanism
```

## Investigation Findings

### Why `script` fails

- `script -qec "cmd" /dev/null` needs to allocate a PTY via `openpty()`/`forkpty()`
- The systemd service has no controlling terminal (`/dev/tty` unavailable)
- Node.js `child_process.spawn` with `stdio: "pipe"` connects stdin/stdout to pipes, not a PTY
- The `script` process can technically still allocate a PTY, but the wrapped CLI sees stdin as a pipe and refuses interactive mode

### Why `node-pty` works

- `node-pty` calls `forkpty()` directly via a native C addon
- The spawned process gets a real PTY device (`/dev/pts/N`) as stdin/stdout/stderr
- `isatty(0)` returns true inside the spawned process
- Works identically from a terminal session or from a systemd service
- Used by VS Code terminal, Hyper, and other production terminal emulators

### Alternatives considered and rejected

| Alternative | Why rejected |
|---|---|
| `expect`/`unbuffer` | Not installed; adds system dependency; still relies on `script` internally |
| `socat` with PTY | Not installed; complex setup; fragile |
| Direct API (no CLI) | Codex and Claude have NO REST API for usage data |
| Non-interactive CLI flag | Neither CLI has one for usage |
| Parse config/cache files | No usage data stored locally |

### node-pty specifics

- **Package**: `node-pty` (npm), maintained by Microsoft
- **Native compilation**: Requires `build-essential` + `python3` (for node-gyp)
- **Prebuilds**: Published for common platforms (Linux x64 glibc)
- **API**: `pty.spawn(command, args, options)` returns IPty with `onData`, `onExit`, `write`, `kill`
- **Output**: Single stream (stdout+stderr combined) — PTY doesn't separate them
- **License**: MIT

### Step 1.2: Rewrite interactive-command.ts

**File**: `apps/backend/src/providers/shared/interactive-command.ts`

Current implementation (to be replaced):
```ts
// OLD: uses script -qec
export async function runInteractiveCommand(
  command: string, args: string[] = [], options: InteractiveCommandOptions = {},
): Promise<SubprocessResult> {
  const scriptCommand = buildScriptCommand(command, args);
  return await runSubprocess("script", ["-qec", scriptCommand, "/dev/null"], { ... });
}
```

New implementation:
```ts
import { type SubprocessResult, SubprocessError } from "../../utils/subprocess.js";

export interface InteractiveCommandOptions {
  cwd?: string;
  env?: NodeJS.ProcessEnv;
  input?: string;
  timeoutMs?: number;
}

export class PtyUnavailableError extends Error {
  constructor() {
    super(
      "node-pty is not available. Install build-essential and rebuild:\n" +
      "  sudo apt install build-essential python3\n" +
      "  pnpm install"
    );
    this.name = "PtyUnavailableError";
  }
}

export async function runInteractiveCommand(
  command: string,
  args: string[] = [],
  options: InteractiveCommandOptions = {},
): Promise<SubprocessResult> {
  // Dynamic import: if node-pty failed to compile, give clear error
  let pty: typeof import("node-pty");
  try {
    pty = await import("node-pty");
  } catch {
    throw new PtyUnavailableError();
  }

  return new Promise<SubprocessResult>((resolve, reject) => {
    const startedAt = Date.now();
    let output = "";
    let settled = false;

    const term = pty.spawn(command, args, {
      cols: 120,
      rows: 30,
      cwd: options.cwd,
      env: { ...process.env, ...options.env },
    });

    const finish = (callback: () => void) => {
      if (settled) return;
      settled = true;
      callback();
    };

    term.onData((chunk: string) => {
      output += chunk;
    });

    const timeout = setTimeout(() => {
      term.kill();
      finish(() =>
        reject(
          new SubprocessError(`Subprocess timed out: ${command}`, {
            command,
            args,
            stdout: output,
            stderr: "",
            durationMs: Date.now() - startedAt,
          }),
        ),
      );
    }, options.timeoutMs ?? 15_000);

    // Delay input: CLI needs time to initialize its prompt
    if (options.input) {
      setTimeout(() => {
        if (!settled) {
          term.write(options.input!);
        }
      }, 200);
    }

    term.onExit(({ exitCode }) => {
      clearTimeout(timeout);
      finish(() =>
        resolve({
          command,
          args,
          exitCode,
          stdout: output,
          stderr: "",
          durationMs: Date.now() - startedAt,
        }),
      );
    });
  });
}

// Keep these unchanged — used by all parsers
export function stripAnsi(text: string): string {
  return text.replace(/\u001b\[[0-9;?]*[ -/]*[@-~]/g, "");
}

export function normalizeLineEndings(text: string): string {
  return text.replace(/\r\n/g, "\n").replace(/\r/g, "\n");
}
```

**Key design decisions**:
- **Dynamic import**: `await import("node-pty")` catches compilation failures gracefully
- **200ms input delay**: Interactive CLIs need time to render their prompt before accepting input
- **Settled guard**: Prevents double-resolve from timeout + onExit race
- **Same signature**: Callers don't need changes
- **Combined stdout**: PTY merges stdout+stderr; `stderr: ""` is always empty

## Risks

| Risk | Mitigation |
|---|---|
| node-pty fails to compile (missing build-essential) | `PtyUnavailableError` with clear install instructions; doctor check in Etapa 2 |
| PTY output differs from `script` output (ANSI sequences, line endings) | `stripAnsi()` and `normalizeLineEndings()` already handle this |
| Input timing too fast (200ms not enough) | Configurable via `options.inputDelayMs`; codex/claude parsers are robust to partial output |
| node-pty adds ~2MB to node_modules | Acceptable for a desktop tool |
| Tests need updating (mock `runInteractiveCommand` instead of `runSubprocess`) | Update test mocks in Etapa 1 implementation |
