## Surface Contract

### Indicator

| State | Icon | Label | Contract |
|-------|------|-------|----------|
| Loading | `view-refresh-symbolic` | `Refreshing` | Keep the label short; do not show provider-specific counts while loading |
| Healthy data loaded | `emblem-ok-symbolic` | `{healthyCount}/{providerCount} ok` | Replace the current generic provider count with an aggregate health ratio |
| Provider issues present | `dialog-warning-symbolic` | `{issueCount} issue` or `{issueCount} issues` | Use warning state for any degraded, error, or unavailable provider unless the whole backend failed |
| Backend/service failure | `dialog-error-symbolic` | `Service` | Reserve the strongest state for extension-wide/backend-wide failure |
| No data | `dialog-information-symbolic` | `No data` | Calm neutral fallback; never show `0 providers` |

Rules:
- Keep the top-bar surface aggregate-only. No provider icons or per-provider strips in the panel.
- Cap the visible label to one short token pair. The indicator is for triage, not explanation.
- Preserve the existing state-driven rebuild pattern from `indicator.js` and `view-model.js`.

### Expanded Menu

| Section | Content | Contract |
|---------|---------|----------|
| Summary row | One compact aggregate line | Replace the current title-plus-body prose block with a single concise summary row |
| Provider list | One structured row per provider | Preserve backend/config order; do not reorder rows for styling reasons |
| Details area | Last updated, backend diagnostics, suggested command | Keep visually secondary below the provider list and separator |
| Action row | Refresh Now / Refreshing... | Keep as the final action; disable while a refresh is in flight |

Rules:
- The provider list is the primary scan target, the summary row is secondary context, and the details/action rows are tertiary support.
- Remove paragraph-like status prose from the top of the menu.
- Section labels should be brief and optional; if used, keep them single-line and muted.
- The menu should fit quick scanning in one open-close cycle without forcing reading of wrapped paragraphs.

### Provider Row Anatomy

| Area | Contract |
|------|----------|
| Left slot | 16px provider identity icon inside a neutral 24px slot with 8px gap to content |
| Header line | Provider name at `14px/600`; trailing status label at `12px/400` with optional symbolic status glyph |
| Primary metric | Compact quota line plus 4px progress bar directly beneath the header |
| Secondary line | One short metadata line only; prefer reset window or concise issue text, not raw diagnostics paragraphs |

Rules:
- Healthy rows show: provider identity, quota numbers, progress bar, reset label when available.
- Degraded/error/unavailable rows show: provider identity, status label, progress bar only if quota data exists, and one short issue summary.
- Remove inline `Suggested command:` text from provider rows. Commands belong in the details area.
- Remove inline `Source:` text from healthy rows. Source is secondary and only belongs in details when it explains a failure.
- Remove per-row `Updated ... ago` from the default row layout. Use the details area for update recency.

### Progress Bar Contract

| Property | Value |
|----------|-------|
| Height | 4px |
| Radius | 999px |
| Track color | Secondary neutral surface |
| Fill color | Provider accent when quota exists and state is healthy/degraded |
| Error/unavailable state | Keep a neutral track; do not fill the bar red unless actual quota data is present |

Rules:
- Numeric quota must always accompany the bar; the bar is never the only signal.
- Clamp fill between `0%` and `100%`.
- Progress bars should read as compact operational indicators, not decorative charts.

### Diagnostics and Errors

| Situation | Contract |
|-----------|----------|
| Provider-specific issue | Show one short issue summary in the row, with the actionable command moved to Details |
| Backend failure | Show one backend error line in Details plus the doctor command |
| Missing prerequisite | Prefer human-readable labels such as `Missing secret-tool` or `Auth needed`; avoid raw exception text in the main scan path |
| Refresh in flight | Keep current single-flight gating; label action `Refreshing...` and disable the action |

Rules:
- Status meaning must never rely on color alone. Every degraded/error state needs text and/or iconography.
- Keep diagnostics readable from the desktop surface, per Phase 5, but visually secondary to quota monitoring.
- Do not add a separate diagnostics-only view in this phase.

### Asset Strategy

| Asset | Contract | Source |
|-------|----------|--------|
| Claude icon | Reuse `qbar/icons/claude-code-icon.png` in provider rows | `07-CONTEXT.md` |
| Codex icon | Reuse `qbar/icons/codex-icon.png` in provider rows | `07-CONTEXT.md` |
| Copilot icon | Use a neutral in-extension fallback badge/lettermark until qbar ships a dedicated Copilot asset; do not fetch external assets | default; no Copilot icon exists in `qbar/icons` |
| Packaging | Copy referenced assets into the GNOME extension package; do not resolve icons from the qbar workspace at runtime | `07-CONTEXT.md` integration note |

### Motion and Interaction

Rules:
- Motion stays minimal. No animated progress fills, no row reordering, no decorative transitions.
- Hover/focus feedback should come from GNOME-native surface changes only.
- Provider rows remain informational in this phase; the only required menu action is refresh.
- Respect GNOME keyboard/focus behavior and existing popup semantics.
